## 一、项目概述

AiCall 是一套面向医疗机构的在线远程会诊平台，覆盖**患者端（移动 H5）、医生端（PC 后台）、管理端（PC 后台）**三大角色。采用前后端分离架构，后端 Spring Boot 3.3.6 + Java 17，前端 Vue 3 + TypeScript Monorepo，集成 LangChain4j 实现 **AI 预问诊、病历摘要生成、OCR 识别、报告生成与质控评分、RAG 知识库、随访问卷自动生成与分析、会诊纪要生成**等 9 项 AI 能力，并接入腾讯 TRTC 实现实时音视频会诊。

## 四、核心功能实现详解

### 4.1 三端认证 + Redis 会话管理

**涉及文件：** `JwtTokenProvider.java`、`JwtAuthenticationFilter.java`、`SecurityConfig.java`、`RedisSessionService.java`

**具体实现：**

```
1. JwtTokenProvider
   - JJWT 生成 Token，Payload 字段：sub(userId) + username + role(PATIENT/DOCTOR/ADMIN)
   - 签名密钥随机生成，有效期 6 小时
   - validateToken() 验证签名 + 过期时间

2. RedisSessionService（核心难点已解决）
   - 问题：最初使用 RedisTemplate<String, Object> 配置 GenericJackson2JsonRedisSerializer
           该序列化器写入 Redis 时在 JSON 中嵌入 @class 类型元数据，但反序列化
           Map<String, Object> 时因缺少 @class 字段抛出 SerializationException
   - 解决：改用 StringRedisTemplate + 注入 ObjectMapper 手动序列化/反序列化
         写入：String json = objectMapper.writeValueAsString(data) → SET key json EX 1800
         读取：String json = GET key → objectMapper.readValue(json, new TypeReference<>(){})
         好处：不依赖 @class 类型元数据，纯 JSON 可读可调试
   - Key 格式：session:PATIENT:123 / session:DOCTOR:456 / session:ADMIN:789
   - TTL：每次 refreshSession() 成功后续期 30 分钟

3. JwtAuthenticationFilter（OncePerRequestFilter 子类）
   - 从 Authorization: Bearer xxx 头解析 Token
   - JWT 验证通过后立即调用 redisSessionService.refreshSession(userId, role)
   - Session 不存在 → 不设置 SecurityContext → 后续 SecurityConfig 判定未认证 → 返回 401
   - Session 存在 → 刷新 TTL → 设置 UsernamePasswordAuthenticationToken 到 SecurityContext

4. SecurityConfig
   - SessionCreationPolicy.STATELESS（无状态，不依赖 HttpSession）
   - CSRF 关闭（REST API 不需要）
   - URL 规则：登录/注册 permitAll，其余 authenticated()
   - 自定义 AuthenticationEntryPoint：返回 JSON {"code":401,"message":"未登录"}（而非 Spring 默认 302 重定向）
   - 自定义 AccessDeniedHandler：返回 JSON {"code":403,"message":"无权限"}
```

**量化成果：** 支持 3 种角色独立认证体系，JWT 6h + Redis 30min 双重过期，无状态设计支持水平扩展。

### 4.2 AI 预问诊 — 多轮对话分诊护士

**涉及文件：** `PreDiagnosisService.java`（~130 行）

**具体实现：**

```
核心设计：
1. System Prompt 角色定义：
   "你是一位专业的分诊护士。按顺序追问：主诉→现病史→既往史→过敏史，
    每次只问一个问题，已收集到的信息不重复追问，
    所有信息收集完毕时回复必须以 [DIAGNOSIS_COMPLETE] 开头"

2. 对话存储策略（Redis）：
   - Key: chat:consultationId，TTL: 2 小时
   - 消息格式：system:xxx / user:xxx / ai:xxx（冒号分隔前缀）
   - 上限 20 条：超过时保留 SystemMessage，从 index=1 开始删除旧消息
   - 序列化方式：toRawList() → List<String> → redisTemplate.opsForValue().set()
   - 反序列化方式：loadRaw(key) 返回 List<String> → toChatMessages() 按前缀重建
     SystemMessage / UserMessage / AiMessage 对象

3. 每次请求的流程：
   chat(consultationId, userMessage):
     1. loadRaw(key) → List<String> 从 Redis 读取
     2. toChatMessages(rawList) → List<ChatMessage> 含历史对话上下文
     3. 追加当前用户的 UserMessage
     4. trimMessages() → 若超 20 条，保留 SystemMessage + 最近 19 条
     5. chatLanguageModel.generate(messages) → AI 回复
     6. 追加 AiMessage → saveRaw(key, toRawList(messages)) 写回 Redis
     7. 检测回复中是否含 [DIAGNOSIS_COMPLETE] 标记 → 返回 finished=true
     8. finished=true 时前端自动触发 generateSummary
```

**量化成果：** 单次问诊平均 4-6 轮对话即可收集完整病情信息，Redis 对话缓存助力上下文持久化。

### 4.3 AI 病历摘要生成 — Prompt Engineering

**涉及文件：** `SummaryService.java`

**具体实现：**

```java
// Prompt 模板（硬编码医疗文书规范）
private static final String SUMMARY_PROMPT = """
    你是一位资深医学文书专家。请将以下患者的预问诊信息整理为结构化的病情摘要。
    要求：按以下结构输出，如果某项信息未提及，标注"未提供"
    格式：
    【主诉】...
    【现病史】...
    【既往史】...
    【过敏史】...
    【初步印象】...（注明仅供参考）

    患者信息：
    %s
    """;

public String generateSummary(String content) {
    String prompt = String.format(SUMMARY_PROMPT, content);
    return chatLanguageModel.generate(prompt);  // 同步调用 LLM
}
```

**三种触发路径：**
| 路径 | 输入 | 调用方 |
|---|---|---|
| 对话模式 | 完整多轮对话文本（患者：... 护士：...） | `ConsultationService.generateSummaryFromChat()` |
| 表单模式 | "主诉：xxx\n起病时间：xxx\n症状描述：xxx\n..." | `ConsultationService.formSubmit()` |
| 挂号模式 | "主诉：" + chiefComplaint（只有一行） | `RegistrationService.register()` |

挂号模式特殊处理：AI 只基于主诉生成摘要，其余章节标注"未提供"，医生后续可补充。

**量化成果：** 统一 Prompt 模板驱动 3 种入口，AI 生成标准五维结构化摘要。

### 4.4 医疗影像 OCR + 向量检索全链路

**涉及文件：** `OcrService.java`、`EmbeddingService.java`、`ConsultationService.uploadFile()`

**具体实现：**

```
上传链路（ConsultationService.uploadFile）：
1. 用户上传文件 → MinIO 存储（bucket: uploads，路径: consultation/{id}/{uuid}.ext）
2. 写入 consultation_upload 表（fileName, fileUrl, fileSize, fileType）
3. 异步线程启动 OCR（避免阻塞用户请求）：
   new Thread(() -> {
     byte[] bytes = file.getBytes()
     String base64 = Base64.getEncoder().encodeToString(bytes)
     String ocrResult = ocrService.recognize(base64)       // ← AI 视觉模型
     uploadMapper.updateOcrResult(uploadId, ocrResult)      // 回写 DB
     embeddingService.storeDocument(consultationId, id, ocrResult)  // ← 向量化存储
   }).start()

OCR 实现（OcrService.recognize）：
- 模型：Qwen3.6-35B-A3B（SiliconFlow 视觉模型）
- 输入：Base64 编码的图片
- Prompt：要求输出结构化 JSON
  {"type":"化验单/影像报告/病理报告",
   "items":[{"name":"白细胞","value":"6.8","unit":"10^9/L","status":"正常"}]}

Embedding 实现（EmbeddingService.storeDocument）：
- 模型：BGE-M3（SiliconFlow 嵌入模型，768 维）
- 向量库：Qdrant，Collection: consultation_document，距离: Cosine
- 存储内容：OCR 结果文本 → 向量化 → Qdrant 存储（含 consultationId / uploadId 元数据）
- 用途：后续医生可通过语义检索找到相关历史病历
```

**量化成果：** 支持 JPG/PNG/PDF 格式，OCR 输出结构化 JSON，向量检索支持语义相似度查询。

### 4.5 AI 报告生成 + 三维质控评分

**涉及文件：** `ReportGenerateService.java`、`QcService.java`

**具体实现：**

```
报告生成（ReportGenerateService.generateReport）：
数据采集阶段（5 个数据源）：
1. Consultation 表：主诉（chiefComplaint）+ AI 摘要（medicalSummary）
2. ConsultationUpload 表：所有已上传文件的 OCR 结果拼接
3. PreDiagnosisService.getHistory()：预问诊对话历史（从 Redis 读取）
4. ReportTemplate 表：按 department 匹配科室报告模板

Prompt 组装 → chatLanguageModel.generate(prompt) → 返回结构化报告 JSON

质控评分（QcService.checkQuality）：
三维评价体系：
┌──────────────┬────────┬──────────────────────┐
│ 维度          │ 权重   │ 评价标准              │
├──────────────┼────────┼──────────────────────┤
│ completeness  │ 30%    │ 各章节是否齐全        │
│ standard      │ 40%    │ 术语准确、书写规范    │
│ consistency   │ 30%    │ 逻辑一致、无矛盾      │
└──────────────┴────────┴──────────────────────┘

计算：totalScore = completeness×0.3 + standard×0.4 + consistency×0.3

JSON 解析容错策略（应对 LLM 非确定性输出）：
1. 从 AI 回复中提取 {} 块（兼容包裹在 ```json 中的情况）
2. objectMapper.readTree(json) → 逐字段读取
3. 任意字段缺失 → 兜底默认值 80 分 ← 保证流程不中断

通过阈值：≥ 60 分 → status=1（通过）→ 医生可签名
          < 60 分  → status=2（不通过）→ 医生需修改重新提交
```

**量化成果：** 报告生成整合 5 类数据源，质控三维打分 + JSON 容错解析防 LLM 输出不稳定。

### 4.6 RAG 医学知识库

**涉及文件：** `KnowledgeService.java`、`TextParserService.java`

**具体实现：**

```
文档入库流程：
1. 文件上传 → PDFBox（PDF）/ POI（Word）/ 直接读取（TXT）→ 提取纯文本

    [Chunk1 内容]
    [Chunk2 内容]
    [Chunk3 内容]
    用户问题：{question}

4. DeepSeek-V3.2 生成回答 + 引用来源标注
```

**量化成果：** 支持 PDF/Word/TXT 三种格式，RAG 检索增强回答带出处标注。

### 4.8 TRTC 实时音视频 + AI 会诊纪要

**涉及文件：** `LiveRoomController.java`、`MinutesService.java`、`WebSocketHandler.java`

**具体实现：**

```
1. UserSig 本地自研签名（关键难点）：
   - 不依赖腾讯云 Java SDK（SDK 太重，引入额外复杂度）
   - 参考 TLSSigAPIv2 规范，使用 Hutool DigestUtil.hmacSha256() 手工实现
   - 参数：SDKAppID=1600143811, SecretKey, userId, expireTime(1小时)
   - 算法：HMAC-SHA256(sigDocument, secretKey) → Base64 → zlib compress → Base64URL

2. WebSocket 实时字幕：
   - WebSocketHandler：管理连接 Map<roomId, List<Session>>
   - 消息格式：{roomId, userId, userName, content, timestamp}
   - 广播：向房间内所有 Session 发送
   - 持久化：每次字幕写入 live_subtitle 表（备后续查询和纪要生成）

3. AI 纪要生成（MinutesService.generateMinutes）：
   - 会诊结束 → 拉取所有 live_subtitle 记录
   - 拼接为：医生A：xxx\n医生B：xxx\n...
   - Prompt 驱动生成 5 个章节：参与专家 → 讨论要点 → 诊断意见 → 治疗建议 → 随访计划
   - 存入 consultation.minutes + status=6
```

**量化成果：** 自研 UserSig 不依赖腾讯 SDK，字幕实时同步 + 持久化，AI 自动生成结构化会诊纪要。

### 4.9 患者挂号流程 — 从选医生到 AI 摘要一气呵成

### 4.10 三层状态机 + 统一异常处理

```
会诊状态流转（Consultation.status 字段）：
0  草稿        → 患者创建
1  摘要已生成   → AI 自动
2  专家确认中   → 患者支付 / 管理员指派
3  已排期       → 医生确认接诊
4  报告已生成   → AI 生成报告
5  已完成       → 医生签名
6  会诊中       → TRTC 房间进行中
7  已取消       → 管理员取消
8  已退回       → 医生退回

全局异常处理机制：
- GlobalExceptionHandler（@RestControllerAdvice）
  → BusinessException → Result.fail(code, message)
  → MethodArgumentNotValidException → Result.fail(400, fieldErrors)
  → Exception → Result.fail(500, "系统异常")

- @Log 注解 + LogAspect（@Around）
  → 自动记录方法入参/出参/耗时
```
