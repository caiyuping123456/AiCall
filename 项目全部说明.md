# AICall 在线MDT多学科会诊平台 — 项目全功能说明

## 项目概述

AICall 是一个面向医院的 AI 辅助多学科远程会诊（MDT）平台。患者通过移动端 H5 发起会诊、与 AI 预问诊对话、上传病历资料；医生通过桌面端接诊、在线音视频会诊、生成专业报告；管理员通过后台面板进行医生管理、会诊监控和数据分析。整个流程由 LLM（大语言模型）提供 AI 问诊、病情摘要、OCR 识别、报告质控和随访问卷等智能辅助。

**技术栈概览：**
- 后端：Java 17 + Spring Boot 3.3 + MyBatis + MySQL + Redis + RabbitMQ + MinIO + LangChain4j + WebSocket
- 前端：Vue 3 + TypeScript + Vite 5 + pnpm monorepo（用户端 Vant 4，医生端/管理员端 Element Plus）
- AI 模型：DeepSeek-V3.2（对话/摘要/质控/随访）、BAAI/bge-m3（向量嵌入）、Qwen3.6-35B-A3B（视觉/OCR），通过 SiliconFlow API 接入
- 视频：腾讯 TRTC Web SDK，支持音视频通话和屏幕共享

## 一、后端架构与功能实现

### 1.1 认证与鉴权体系

**原理：** 采用无状态 JWT（JSON Web Token）认证方案。用户登录成功后，服务端生成一个包含用户 ID、角色信息的 Token 返回前端；前端在每次请求的 Authorization 头中携带该 Token；后端通过过滤器拦截请求，解析 Token 后将用户身份注入 Spring Security 的安全上下文。

**实现方式：**
- `JwtTokenProvider` 负责 Token 的生成、解析和校验。Token 的 payload 中包含 userId（主键）和 role（角色：PATIENT / DOCTOR / ADMIN），有效期为 24 小时。
- `JwtAuthenticationFilter` 继承 `OncePerRequestFilter`，在每个请求到达 Controller 前从请求头提取 Token，调用 JwtTokenProvider 验证后构造 Authentication 对象，将 userId 设为 principal，role 添加为 `ROLE_` 前缀的权限标识。
- `SecurityConfig` 将 `/user/auth/**`、`/doctor/auth/**`、`/admin/auth/**`、`/ws/**` 标记为公开访问，其余业务接口按路径模式区分权限（如 `/doctor/**` 需要 DOCTOR 角色）。
- 患者密码采用 MD5 + 32 位随机盐存储；医生和管理员密码采用 BCrypt 加密。
- Controller 方法中通过 `Authentication auth` 参数或 `SecurityContextHolder.getContext().getAuthentication()` 获取当前登录用户，其 `auth.getPrincipal()` 返回 userId（Long 类型）。

### 1.2 患者端核心流程（User Flow）

**业务流程：** 注册/登录 → 选择预问诊方式 → AI 对话问诊或表单填写 → 生成病情摘要 → 上传病历资料 → 选择会诊类型 → 支付 → 提交会诊。

**实现方式：**
- `UserAuthService` 处理注册和登录。注册时随机生成 32 位盐值，用 `MD5(password + salt)` 存储密码；登录时查库取盐后做同样计算再比对。登录成功后调用 JwtTokenProvider 签发 Token。
- `ConsultationService` 管理会诊的完整生命周期：创建草稿（status=0）、更新状态、计算费用（单学科 500 元 / MDT 1500 元）、支付（生成支付单并更改 status 为 2）。
- `PreDiagnosisService` 实现 AI 对话问诊。内部维护一个基于 LangChain4j 的对话链，将患者的每条消息连同历史对话一起发给 ChatLanguageModel（DeepSeek-V3.2），模型以分诊护士的角色引导患者描述病情、补充细节。
- `SummaryService` 将预问诊对话或表单内容整理为结构化病情摘要。通过预设的医学文书提示词，要求模型按"主诉、现病史、既往史、过敏史、初步印象"五个维度输出中文内容。
- `EmbeddingService` 将上传文件的 OCR 识别结果通过 EmbeddingModel（BGE-M3）向量化后存入 Qdrant 向量数据库，供后续检索增强生成（RAG）。
- `OcrService` 使用 Qwen3.6 视觉模型对上传的医学影像/化验单进行 OCR 文字识别，输出结构化文本。

---

### 1.3 医生端核心流程（Doctor Flow）

**业务流程：** 登录 → 工作台查看待办 → 确认/拒绝接诊 → 查看患者资料和 AI 摘要 → AI 生成报告 → 编辑报告 → 提交质控 → 签发报告。

**实现方式：**
- `DoctorAuthService` 处理医生登录。医生账号由管理员预创建（用户名格式：`doctor_` + 随机 6 位字符），密码用 BCrypt 存储。登录后签发 Token 并返回医生基本信息。
- `DoctorConsultationService` 提供工作台聚合查询（待审核数、报告编辑数、待质控数、近期会诊列表）、会诊列表筛选、会诊详情查看（含患者信息、上传文件、AI 摘要、对话记录）、接诊确认/拒绝。
- `ReportGenerateService` 根据会诊信息（患者主诉、AI 摘要、对话记录、OCR 结果）调用 ChatLanguageModel 生成结构化专业报告，包括诊断意见、治疗方案、用药建议、随访建议等章节。
- `QcService` 对医生提交的报告内容进行 AI 质控评分。模型分别从完整性、规范性、一致性三个维度打分（0-100），并输出具体问题清单。总分低于阈值的报告需要修改后重新提交。
- `DoctorScheduleMapper` 管理医生的排班记录，支持按日期筛选的新增/删除。

---

### 1.4 管理员端核心流程（Admin Flow）

**业务流程：** 登录 → 数据仪表盘 → 医生管理（增删改查、排班） → 会诊管理（列表、详情、取消） → 录像回放和会议纪要。

**实现方式：**
- `AdminService` 处理管理员登录（BCrypt 验证），生成 JWT。
- `AdminDashboardService` 聚合多个 Mapper 的统计查询：从 consultation 表按 status 分组统计各类会诊数量，按 department 分组统计科室分布，按日统计创建趋势；从 payment_order 表汇总收入金额。前端使用 ECharts 渲染饼图、柱状图、折线图。
- `AdminDoctorService` 提供医生的 CRUD 操作：创建时自动生成用户名（`doctor_` + 随机串），支持按姓名/科室/手机号搜索，分页列表，详情展示含排班信息，支持启用/禁用状态切换。排班子功能支持按日期筛选的增删。
- `AdminConsultationService` 提供会诊列表查询（按状态筛选、按患者姓名/会诊编号搜索、分页）、详情查看（含指派医生、上传资料、会议纪要、录像清单）、取消会诊（填写原因后 status → 7）。

### 1.5 音视频会诊系统（Live Consultation）

**原理：** 基于腾讯 TRTC（实时音视频）构建在线会诊室。后端负责房间管理、鉴权签名生成、WebSocket 实时字幕转发；前端通过 TRTC Web SDK 建立 P2P 音视频连接，浏览器 Web Speech API 进行语音识别，WebSocket 广播字幕。会诊结束后，LLM 根据字幕记录自动生成会议纪要。

**实现方式：**

**房间管理（LiveRoomService）：**
- 创建房间时生成 6 位随机数字作为 TRTC 房间号，状态初始为 0（待开始）。每个咨询（consultation）对应一个房间。
- 进入房间时将状态改为 1（进行中）并记录开始时间；结束房间时状态改为 2（已结束）并记录结束时间。
- UserSig（用户签名）是 TRTC 鉴权的核心。开发模式下 `sdkAppId=0` 返回 mock 签名；正式模式下使用 HMAC-SHA256 对 `sdkAppId|userId|roomId|expireTime` 进行签名后 Base64 编码。

**WebSocket 字幕路由（WebSocketHandler）：**
- 按 consultationId 分组管理 WebSocket 会话。每个会话注册进对应的房间分组（ConcurrentHashMap）。
- 参与者发送 `join` 消息加入房间，发送 `leave` 消息退出，退出时自动移出分组并广播离开通知。
- 前端语音识别结果通过 WebSocket 发送 `subtitle` 消息，后端将其持久化到 live_subtitle 表，然后广播给同一 consultationId 下的所有其他会话。这样所有参会者都能看到实时字幕。

**AI 会议纪要（MinutesService）：**
- 会诊结束后，从 live_subtitle 表按 roomId 查询所有字幕记录，按时间升序拼接成"用户名：内容"格式的会议记录。
- 将完整记录通过预设提示词发给 ChatLanguageModel，要求按"参与专家、讨论要点、诊断意见、治疗建议、随访计划"结构输出 Markdown 纪要。
- 生成的纪要存入 consultation 表的 minutes 字段，同时将 consultation 状态更新为 6（已完成）。

---

### 1.6 随访系统（Follow-up System）

**原理：** 会诊完成后自动为患者创建多期随访计划，每期包含 AI 生成的个性化问卷；到期时通过定时任务自动发送通知；患者填写后 AI 分析回答内容，异常情况标记并通知医生。

**实现方式：**

**随访创建（FollowUpService.createFollowUps）：**
- 会诊结束时（LiveRoomController.endRoom）自动触发。读取 sys_config 表的 `follow_up_days` 配置（默认 "3,7,30"），为每个天数创建一条随访记录（status=0），并通过 ChatLanguageModel 为每期生成个性化问卷。
- 问卷生成提示词包含患者主诉和 AI 摘要，要求模型按 JSON 格式输出 5-8 个问题（支持单选 radio、多选 checkbox、文本 text 三种类型）。

**答案提交与 AI 分析：**
- 患者提交回答后，答案以 JSON 字符串存入 answer 字段，异步触发 AI 分析。
- AI 分析提示词包含原始问卷和患者回答，要求模型判断是否异常并输出 JSON（含 abnormal 布尔值、level 异常等级、summary 分析总结、suggestion 处理建议）。
- 若判定为异常，将随访状态设为 3（异常告警），并通过 NotificationService 通知相关医生。

**定时调度（FollowUpScheduler）：**
- 使用 Spring `@Scheduled` 注解，每天上午 9 点执行。
- SQL 通过 `DATE_ADD(DATE(c.end_time), INTERVAL f.plan_day DAY) = today` 匹配当天到期的随访记录，检查 status=0 的待发送记录，更新 status=1 并调用 NotificationService 发送通知。

---

### 1.7 评价系统（Evaluation System）

**原理：** 会诊完成后自动创建评价记录；患者可对医生和服务进行星级评分并撰写文字评价；评价结果关联到会诊详情供医生和管理员查看。

**实现方式：**
- `EvaluationService.createEvaluation` 在会诊结束时被调用，插入一条空白评价记录（consultationId + patientId）。
- 患者提交评价时，通过 `updateScore` 更新医生评分（1-5）、服务评分（1-5）和可选文字评价。已评价的记录不能重复提交。
- ConsultationMapper 的 findById 查询通过 LEFT JOIN evaluation 将评价信息（评分和评语）一并查出，在会诊详情中展示。

---

### 1.8 通知系统（Notification System）

**原理：** 统一的通知服务，支持站内信（持久化到数据库）和 WebSocket 实时推送两种通道；患者和医生各有独立的通知列表；通知按类型区分（随访提醒、评价邀请、异常告警）。

**实现方式：**
- `NotificationService.send` 接收用户类型、用户 ID、标题、内容、通知类型列表。type=2 时创建数据库记录（站内信），type=3 时同时创建数据库记录和通过 WebSocketHandler.sendToUser 实时推送。
- WebSocketHandler.sendToUser 通过遍历当前在线会话的 sessionUserId 映射表找到目标用户的所有 WebSocket 连接，向每个打开连接发送 notification 消息。
- 通知列表支持分页查询、未读数量统计、标记已读操作。通知中心前端页面在进入时加载列表，点击后标记为已读。

---

### 1.9 文件存储

**原理：** 使用 MinIO（兼容 S3 协议的对象存储）管理用户上传的病历资料、影像文件、检查报告等。上传后将文件 URL 存入 consultation_upload 表，并通过多线程异步触发 OCR 识别和向量化存储。

**实现方式：**
- `MinioStorageService` 封装 MinIO SDK 的上传操作。文件以 `consultation/{consultationId}/{UUID}.{ext}` 作为对象名存储，保证唯一性和可追溯性。
- 上传成功后，异步线程调用 OcrService 进行文字识别，将 OCR 结果回写到 upload 记录；再调用 EmbeddingService 将文本向量化存入 Qdrant，为后续 RAG 检索提供支撑。

---

### 1.10 消息队列（RabbitMQ）

**原理：** 预备的异步消息基础设施，当前定义了一个通知队列（notification），可由生产者发布消息，未来可由独立消费者处理短信发送、邮件通知等耗时操作。

**实现方式：**
- `RabbitMqConfig` 声明了 NOTIFICATION_QUEUE 队列和路由键。
- `RabbitMqProducer` 提供 `sendNotification(Object message)` 方法，将消息序列化后通过 RabbitTemplate 发布到指定队列。

## 二、前端架构与功能实现

### 2.1 整体架构

前端采用 pnpm monorepo 管理 4 个包：

| 包名 | UI 框架 | 端口 | 用途 |
|------|--------|------|------|
| shared | — | — | 共享的 API 客户端、TypeScript 类型、工具函数 |
| user | Vant 4 | 3000 | 患者移动端 H5 |
| doctor | Element Plus | 3001 | 医生桌面端 |
| admin | Element Plus | 3002 | 管理员后台面板 |

### 2.2 共享包（@aicall/shared）

**原理：** 将所有后端 API 调用、TypeScript 接口定义、通用工具函数集中在一个包中，其他三个包通过 workspace 协议（`workspace:*`）引用。这样每个 API 只需定义一次，各端共享使用。

**实现方式：**
- `request.ts` 封装 Axios 实例，设置 baseURL 为 `/api`（由 Vite 代理转发到 localhost:8080）。请求拦截器从 localStorage 读取 Token 添加到 Authorization 头；响应拦截器自动解包后端返回的 `{ code, message, data }` 结构——成功时返回 `data`，失败时抛出错误供页面捕获。
- 每个 API 模块（consultation.ts、doctor.ts、admin.ts、live.ts、followup.ts、evaluation.ts、notification.ts）通过泛型函数 `get<T>` / `post<T>` / `put<T>` / `del<T>` 定义接口，返回类型明确的 Promise。
- types 目录定义了通用分页结构 `PaginatedResult<T>`（含 list、total、page、size）和登录请求/响应接口。
- utils 提供格式化工具：文件大小、日期时间、手机号和姓名脱敏。

---

### 2.3 患者端（User Package — Vant 4 移动端 H5）

**原理：** 面向患者的移动端 Web 应用（H5），覆盖从注册登录到会诊提交的完整流程，以及会诊后的随访填写、评价提交和通知查看。

**实现方式：**
- **路由与鉴权：** 所有页面在 router 中静态导入（无懒加载）。需要登录的页面设置 `meta.requiresAuth: true`，由 main.ts 中的全局导航守卫检查 localStorage 中的 Token，未登录跳转到 `/login`。
- **状态管理：** 不引入 Pinia/Vuex，每个页面通过 Vue 3 组合式 API 的 `ref` 和 `reactive` 管理本地状态。患者 ID 通过 `localStorage.getItem('patientId')` 获取。
- **会诊流程：** 9 个页面构成的线形流程——Start → Chat/Form → Summary → Upload → SelectType → Pay → Success → Query。每步用 `van-steps` 展示进度，数据通过路由参数 `:id`（consultationId）串联。
- **AI 对话页：** 使用 Vant 的 `van-cell-group` 和 `van-field` 构建聊天气泡界面。患者发送消息后调用 API，AI 回复内容实时追加到消息列表，通过 `nextTick` + `scrollTop` 自动滚动到底部。
- **表单填写页：** 结构化的病史表单（主诉、起病时间、症状描述、既往史、过敏史），支持文本输入和日期选择。
- **文件上传页：** 使用 `van-uploader` 组件调用手机相机或相册选择文件，上传后展示 OCR 识别结果。
- **随访填写页：** 将 API 返回的 JSON 问卷解析为动态表单——radio 问题渲染为 `van-radio-group`，text 问题渲染为 `van-field` textarea。提交时将答案数组 JSON 序列化后发送。
- **评价页：** 使用 `van-rate` 五星评分组件分别评价医生和服务，支持可选文字评语。
- **通知中心：** 使用 `van-cell-group` 展示通知列表，点击后调用标记已读 API。

---

### 2.4 医生端（Doctor Package — Element Plus 桌面端）

**原理：** 面向医生的桌面 Web 应用，提供工作台仪表盘、会诊管理、在线视频会诊、报告编辑质控和通知中心等功能。

**实现方式：**
- **布局与鉴权：** 使用 `MainLayout.vue` 作为外层布局（左侧导航菜单 + 顶部信息栏 + 主内容区）。路由全部懒加载（`() => import(...)`），嵌套在 `/` 路径的 children 下。鉴权方式与患者端相同——检查 Token 后跳转 `/login`。
- **状态管理：** 使用 Pinia 存储医生登录后的 token、doctorId、name、department、title 等信息，localStorage 持久化，页面刷新后自动恢复。
- **工作台：** Dashboard 页调用 `/doctor/workbench` API 聚合待办统计（待审核数、报告编辑数、待质控数）和近期会诊列表。
- **会诊详情页：** 使用 `el-tabs` 组织多个标签页——患者信息（`el-descriptions`）、病情摘要、上传资料表（含 OCR 结果）、问诊记录（聊天气泡）、报告（含签发状态）、随访记录（问卷、回答和 AI 分析）。每个标签页独立加载数据。
- **在线会诊室（ConsultationRoom）：** 
  - 进入页面时先调用 API 获取或创建 LiveRoom，再获取 TRTC UserSig。
  - 动态导入 `trtc-js-sdk`（通过 `import()` 懒加载避免主包体积膨胀），使用 UserSig 初始化 TRTC 客户端并加入房间，发布本地视频流并订阅远端流。
  - 使用浏览器原生的 Web Speech API（`webkitSpeechRecognition`）进行语音识别，设置 `continuous: true`（持续识别）和 `lang: 'zh-CN'`。识别结果通过 WebSocket 发送 subtitle 消息给后端广播。
  - 屏幕共享使用 TRTC SDK 的 screen share 流类型，调用浏览器 `getDisplayMedia` 获取屏幕画面。
  - 底部工具栏提供摄像头/麦克风切换（切换视频和音频流的 mute/unmute）、屏幕共享、结束会诊按钮。
- **报告编辑页：** 富文本风格的报告编辑器，展示 AI 生成的结构化报告，医生可修改各章节内容后提交质控或签发。
- **通知中心：** `el-table` 展示通知列表，支持"标已读"操作。

---

### 2.5 管理员端（Admin Package — Element Plus 桌面端）

**原理：** 面向管理员的桌面 Web 应用，提供全局数据仪表盘、医生管理（含排班）、会诊管理和录像/纪要查看功能。

**实现方式：**
- **布局与鉴权：** 与医生端相同的 `MainLayout` + 懒加载路由模式。左侧菜单含 Dashboard、Doctors、Consultations 三个入口。使用 Pinia admin store 管理登录状态。
- **数据仪表盘（Dashboard）：** 
  - 顶部 4 个统计卡片展示今日会诊数、今日收入、本月完成会诊、活跃医生数。
  - 三个 ECharts 图表：饼图（会诊状态分布）、柱状图（各科室会诊量）、折线图（每日趋势）。
  - 医生工作量排行榜表格（按处理会诊数降序）。
- **医生管理（DoctorList / DoctorDetail）：** 
  - 列表页支持姓名/科室/手机号搜索、分页、状态开关（启用/禁用）。新增/编辑医生通过 `el-dialog` 表单弹窗实现。
  - 详情页展示医生完整信息和排班列表（`el-table`）。排班支持按日期筛选、新增（日期选择器 + 时间选择器）和删除（确认弹窗）。
- **会诊管理（ConsultationList / ConsultationDetail）：**
  - 列表页支持状态筛选（el-select）和关键词搜索，分页展示所有会诊记录。
  - 详情页展示会诊完整信息（`el-descriptions`）、指派医生表、上传资料表。状态非已完成/已取消时显示取消按钮。
  - 会诊纪要以卡片展示 Markdown 文本。
  - 录像列表以表格展示，每条含文件名、时长、文件大小，点击可弹出 `<video>` 标签播放。
- **取消会诊：** 弹出 `el-dialog` 要求输入取消原因，调用 API 后会诊状态变为已取消。

### 2.6 前端关键技术决策

| 决策 | 选择 | 原因 |
|------|------|------|
| Monorepo 工具 | pnpm workspaces | 轻量，原生支持 workspace 协议 |
| 患者端 UI | Vant 4 | 专为移动端设计的 Vue3 组件库 |
| 医生/管理员端 UI | Element Plus | 成熟的桌面端 Vue3 组件库 |

| 图表 | ECharts | 功能全面的数据可视化库 |
| TRTC SDK 加载 | 动态 `import()` | 避免将 580KB 的 SDK 打入主包 |
| 语音识别 | Web Speech API | 浏览器原生支持，无需额外 SDK |
| 实时通信 | 原生 WebSocket | 轻量，满足字幕广播和通知推送需求 |

## 三、数据流总结

### 3.1 会诊全生命周期

```
患者发起 → AI预问诊(o) → 病情摘要(1) → 上传资料 → 计算费用 → 支付(2)
→ 医生确认(3) → 排期(4) → 进入会诊室 → 音视频会诊(5)
→ 结束会诊 → AI纪要(6) → 随访(3/7/30天) → 患者评价
```

括号内数字为 consultation 表的 status 字段值。

### 3.2 核心服务调用链

```
会诊完成 (LiveRoomController.endRoom)
  ├── MinutesService.generateMinutes()       → AI 会议纪要
  ├── FollowUpService.createFollowUps()      → 创建 3 期随访 + AI 问卷
  └── EvaluationService.createEvaluation()   → 创建评价记录

每日定时 (FollowUpScheduler.checkAndSend)
  ├── FollowUpService.sendDueFollowUps()     → 扫描到期随访
  └── NotificationService.send()             → 推送通知

患者提交随访 (FollowUpController.answer)
  └── FollowUpService.submitAnswer()
      └── FollowUpService.analyzeAnswer()    → AI 分析 → 异常则通知医生
```

## 五、总结

AICall 平台围绕 MDT 多学科会诊场景，构建了从患者发起、AI 预问诊、医生接诊、在线音视频会诊、报告生成与质控、到会后随访和评价的完整业务闭环。后端以 Spring Boot + MyBatis 为核心，通过 LangChain4j 集成多个大语言模型实现 AI 智能辅助；前端以 Vue 3 monorepo 架构支撑三类用户终端，兼顾移动端便捷性和桌面端专业度。
