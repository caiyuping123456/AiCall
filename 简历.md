# 项目简历

## AICall AI 在线 MDT 医疗会诊平台

**项目演示地址：** 本地部署演示（前端三端 + Spring Boot 后端 + MySQL/Redis/MinIO/Qdrant）  
**开源地址：** https://github.com/caiyuping123456/AiCall  
**2026.03-2026.05**

**技术栈：** Spring Boot、Spring Security、JWT、MyBatis、MySQL、Redis、RabbitMQ、MinIO、Qdrant、LangChain4j、PDFBox、Vue3、TypeScript、Vite、Pinia、Vant、Element Plus、ECharts、TRTC JS SDK

**项目介绍：**  
AICall 是一个面向远程医疗会诊场景的 AI + MDT 多学科会诊平台，包含用户端、医生端、管理端三端。平台围绕“AI 预问诊、医学资料上传、医生会诊、AI 辅助报告、报告质控、医生签发、随访评价”构建完整会诊闭环，解决传统线上问诊中患者病情描述不结构化、医生整理资料耗时、会诊报告撰写效率低、报告质量缺少自动校验等问题。项目将大模型对话、视觉 OCR、RAG 知识库、向量检索、对象存储、Redis 上下文管理、音视频会诊等能力融合到医疗业务链路中，提升会诊资料整理效率和报告生成质量。

**主要工作：**

- 基于 **LangChain4j + Redis** 实现 AI 预问诊上下文管理，将 AI 设定为分诊护士，通过 Prompt 约束采集主诉、现病史、既往史、过敏史等信息；使用 `chat:{consultationId}` 维护会诊级上下文，限制最多 20 条消息并设置 2 小时 TTL，降低长上下文带来的 token 成本和 Redis 无效占用。

- 基于 **MinIO + 视觉大模型 + Qdrant** 构建医学资料处理链路，患者上传检查报告后先进入 MinIO，数据库仅保存文件元数据；后端调用视觉模型进行 OCR 识别并写回数据库，再通过 Embedding 模型向量化写入 Qdrant，为医生查看、报告生成和语义检索提供结构化数据基础。

- 基于 **Qdrant + RAG** 实现医学知识库问答，管理端支持上传医学文档，系统使用 PDFBox 解析 PDF 文本，并按 500 字切片、50 字重叠生成文本块；用户提问时先进行 TopK 向量检索，再将召回片段拼接到 Prompt 中生成回答，同时返回来源片段，提升医学问答的可追溯性并降低模型幻觉。

- 基于 **大模型 + 报告模板 + AI 质控** 设计会诊报告闭环，医生端根据患者主诉、病情摘要、OCR 检查资料、AI 预问诊记录和科室模板生成结构化报告；报告提交后从完整性、规范性、一致性三个维度进行 AI 评分，并按 30% / 40% / 30% 计算综合分，低分报告自动退回编辑，实现“AI 生成初稿 + 医生审核 + AI 质控 + 医生签发”。

- 基于 **Spring Security + JWT** 实现三类角色认证与接口保护，用户端、医生端、管理端分别维护登录态；密码使用 BCrypt 加密，后端通过 JWT 过滤器解析登录身份，结合角色入口区分患者、医生、管理员的业务访问边界。

- 基于 **Vue3 + TypeScript + Vite** 拆分用户端、医生端、管理端三套前端应用，用户端使用 Vant 适配移动端会诊流程，医生端使用 Element Plus 实现待审核、报告编辑、质控待办等工作台能力，管理端结合 ECharts 展示会诊状态分布、科室分布、收入趋势、近 30 天会诊趋势等运营指标。

- 基于 **TRTC JS SDK** 实现在线音视频会诊能力，后端创建会诊房间并生成 UserSig，前端加入房间、发布本地音视频流并订阅远端流；保留 `remote-video`、`local-video`、`screen-video` 等视频挂载点，并结合字幕记录为后续 AI 生成 MDT 会诊纪要提供数据来源。 

- 基于 **MySQL + Redis + MinIO + Qdrant + RabbitMQ** 进行数据分层设计，MySQL 存储结构化业务数据，Redis 存储短期 AI 对话上下文，MinIO 存储医学资料和知识库文件，Qdrant 支撑语义向量检索，RabbitMQ 预留通知推送和 OCR/Embedding/报告生成等耗时任务异步化能力，使系统不局限于简单 CRUD。

**量化成果：**

- 完成用户端、医生端、管理端 3 个前端应用，覆盖患者发起会诊、医生处理会诊、管理员运营管理等核心场景。
- 实现 10+ 个业务模块，包括用户、医生、管理员、会诊、报告、质控、知识库、文件上传、音视频会诊、随访、评价、通知等。
- 接入 6 类 AI 能力：AI 预问诊、病情摘要、医学 OCR、知识库问答、报告生成、报告质控。
- Redis 预问诊上下文控制在最多 20 条消息，并设置 2 小时 TTL，减少无效上下文带来的内存占用和模型调用成本。
- 医学知识库文档按 500 字切片、50 字重叠，结合 Qdrant TopK 检索提高医学问答上下文召回质量。
- 文件上传配置支持最大 100MB，通过 MinIO 承载医学资料和知识库文件，避免大文件直接进入 MySQL。
- 管理端实现 4 类运营图表：会诊状态分布、科室分布、收入趋势、近 30 天会诊趋势。
- 三端均通过生产构建验证：`pnpm build:user`、`pnpm build:doctor`、`pnpm build:admin`。

# 面试题与标准答案

## 亮点一：LangChain4j + Redis 实现 AI 预问诊上下文

### 面试题 1：AI 预问诊为什么要做上下文管理？

**标准答案：**  
预问诊是一个多轮信息采集过程，AI 需要知道患者前面已经回答了哪些内容，才能判断下一步该追问主诉、现病史、既往史还是过敏史。如果每次只传当前问题，模型容易重复提问或漏采集信息。项目中使用 Redis 按会诊维度保存历史消息，每次调用模型前加载上下文，再追加当前用户输入，实现连续问诊。

### 面试题 2：Redis key 和过期策略怎么设计？

**标准答案：**  
key 按 `chat:{consultationId}` 设计，每个会诊单独维护上下文，避免不同患者或不同会诊互相污染。value 保存用户消息、AI 消息和系统提示转换后的消息列表。同时限制最多 20 条消息并设置 2 小时 TTL，控制 token 成本和 Redis 内存占用。

### 面试题 3：AI 怎么判断问诊完成？

**标准答案：**  
在系统 Prompt 中约定，当主诉、现病史、既往史、过敏史等信息收集完整时，AI 回复必须带 `[DIAGNOSIS_COMPLETE]` 标识。后端解析模型回复时检查该标识，如果存在就认为预问诊完成，并把标识从展示文本中移除，再返回给前端。

## 亮点二：MinIO + OCR + Qdrant 医学资料处理链路

### 面试题 1：患者上传医学资料后的处理流程是什么？

**标准答案：**  
文件上传后先存入 MinIO，MySQL 保存文件名、类型、URL、大小等元数据。随后后端调用视觉模型进行 OCR，识别化验单、影像报告等资料中的文字和指标。OCR 结果写回数据库后，再调用 Embedding 模型生成向量并写入 Qdrant，用于后续语义检索和报告生成。

### 面试题 2：为什么文件不直接存 MySQL？

**标准答案：**  
医学资料通常是图片、PDF 等大文件，直接存 MySQL 会导致表膨胀、备份困难、查询变慢。MinIO 更适合对象存储，MySQL 只保存元数据和访问地址，实现结构化数据和非结构化文件分离，也更接近生产系统架构。

### 面试题 3：OCR 结果为什么要向量化？

**标准答案：**  
OCR 只能把图片转成文本，但如果后续要按语义查找相关资料，关键词匹配不够准确。向量化后可以根据语义相似度检索，例如查询“肝功能异常”时，可以召回包含 ALT、AST、胆红素等相关指标的报告，即使没有完全相同关键词也能召回相关内容。

## 亮点三：Qdrant + RAG 医学知识库问答

### 面试题 1：项目中的 RAG 是怎么实现的？

**标准答案：**  
管理员上传医学文档后，系统解析文档内容并切片，使用 Embedding 模型生成向量存入 Qdrant。用户提问时，先把问题向量化并到 Qdrant 检索相关片段，再把召回内容拼接到 Prompt 中交给大模型生成回答，并返回来源片段。

### 面试题 2：为什么切片要保留 50 字重叠？

**标准答案：**  
固定切片可能把一个完整医学语义切断，例如疾病描述和治疗建议分到两个片段中。保留 50 字重叠可以让相邻文本块共享上下文，降低关键信息被截断的概率，提高召回质量。

### 面试题 3：RAG 如何降低大模型幻觉？

**标准答案：**  
大模型直接回答可能编造不存在的医学知识。RAG 会先从本地知识库检索相关内容，并要求模型基于检索片段回答；如果知识库没有相关信息就如实说明。这样回答有来源依据，也方便医生或用户核对。

## 亮点四：AI 报告生成与质控闭环

### 面试题 1：AI 会诊报告生成用了哪些上下文？

**标准答案：**  
报告生成会组合患者主诉、病情摘要、上传资料 OCR 结果、AI 预问诊记录和科室报告模板。后端将这些信息组织成结构化 Prompt，让模型按固定 JSON 字段输出主诉、现病史、检查所见、诊断意见、分析说明、建议和随访建议。

### 面试题 2：为什么不能 AI 直接签发报告？

**标准答案：**  
医疗场景对安全性和责任要求高，AI 可能出现幻觉或错误建议，不能替代医生决策。项目中 AI 只生成报告初稿，医生必须审核、修改、提交质控并签发，患者才能看到最终报告，体现“AI 辅助，医生负责”的原则。

### 面试题 3：报告质控怎么实现？

**标准答案：**  
报告提交后，大模型从完整性、规范性、一致性三个维度评分，并输出具体问题。系统按 30%、40%、30% 计算综合分，低于阈值则退回编辑，通过后医生才能签发。这样形成生成后的自动校验机制。

## 亮点五：三端分离与协同工作台

### 面试题 1：为什么拆成用户端、医生端、管理端？

**标准答案：**  
三类角色的使用场景不同。用户端关注移动端提交资料和查看报告，医生端关注待办、患者资料、报告编辑和质控，管理端关注医生、患者、科室、知识库和运营数据。拆成三端后职责更清晰，权限边界更明确，也方便独立优化交互体验。

### 面试题 2：为什么用户端用 Vant，医生端和管理端用 Element Plus？

**标准答案：**  
用户端主要面向患者，偏移动端使用，Vant 适合移动表单、底部导航和卡片列表。医生端和管理端偏 PC 工作台，需要大量表格、弹窗、表单、描述列表和图表，Element Plus 更适合后台管理系统。

### 面试题 3：TRTC 会诊室需要注意什么？

**标准答案：**  
TRTC 视频流依赖固定 DOM 容器挂载，所以不能随意改名或删除 `remote-video`、`local-video`、`screen-video`。同时组件卸载时要释放本地流、退出房间、关闭 WebSocket，避免摄像头占用或内存泄漏。

## 亮点六：多基础设施分层设计

### 面试题 1：不同中间件分别负责什么？

**标准答案：**  
MySQL 存结构化业务数据，Redis 存短期 AI 对话上下文，MinIO 存医学资料和知识库文件，Qdrant 做向量检索，RabbitMQ 用于异步通知和任务解耦。不同组件承载不同类型的数据和任务，避免所有能力都堆在 MySQL 中。

### 面试题 2：为什么说这个项目不是简单 CRUD？

**标准答案：**  
项目不仅有用户、医生、科室、会诊等基础业务，还集成了 AI 预问诊、OCR、Embedding、RAG、报告生成、报告质控、音视频会诊、对象存储、向量数据库、Redis 上下文管理、RabbitMQ 异步解耦等能力，涉及 AI、存储、检索、实时通信和多端协同。

### 面试题 3：后续如果优化 RabbitMQ，你会怎么做？

**标准答案：**  
我会把 OCR、Embedding、报告生成等耗时任务改造成 MQ 异步消费。上传文件后接口只保存文件并发送任务消息，消费者负责 OCR 和向量化，同时增加任务状态表、失败重试、死信队列和 WebSocket 进度推送，提高可靠性和用户体验。
